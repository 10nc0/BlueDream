# Your Nyanbook~ 🌈

## Overview
"Your Nyanbook" is a multi-tenant SaaS messaging book designed to securely forward messages from WhatsApp to Discord. It provides robust authentication, permanent message retention via Discord threads, and isolated data storage for each user. The project aims to offer a zero-friction onboarding experience and a highly customizable, responsive interface for managing personal message archives.

## User Preferences
- **Design**: Apple glassmorphism aesthetic with Discord-style message layout
- **Privacy**: Messages sent TO book only (not group monitoring)
- **Retention**: Permanent storage (no deletion of messages)
- **Security**: Multi-user authentication with audit logging
- **Compatibility**: Safari/iPad support (JWT localStorage + proper cookie handling)
- **UX**: Auto-expanding interface, responsive design, user-customizable layout, zero-friction onboarding (no webhook required), progressive disclosure for power features

## System Architecture
The system uses a Node.js backend with Express and a Single Page Application (SPA) frontend, featuring an Apple glassmorphism design and a Discord-style two-pane layout. It supports real-time updates and is responsive across devices.

**UI/UX Decisions:**
- **Adaptive Layout**: Desktop offers a resizable sidebar and header; mobile features automatic layout, harmonized header, and a floating action zone.
- **Touch Interactions**: Optimized for iPhone with tap-to-zoom, swipe navigation, auto-hide elements on scroll, 48px touch targets, and momentum scrolling. Cat animation is interactive on dashboard but non-interactive on auth pages.
- **Responsive Components**: UI components adapt for mobile/desktop contexts.

**Technical Implementations:**
- **Authentication**: Email/password authentication using JWT tokens, role-based access control, and isolated user data storage. Email = tenant identity (1:1 mapping via core.user_email_to_tenant).
- **Database**: PostgreSQL with multi-tenant architecture, ensuring data separation per user via isolated schemas.
- **Book Registry**: **Centralized global registry** (core.book_registry) provides O(1) join code lookups, eliminating N-schema loops (26+ queries → 1 query per WhatsApp message). Stores book metadata (join_code, fractal_id, tenant linkage, status, multi-outpipe configuration) with dynamic indexing on join_code (UNIQUE), tenant_schema, fractal_id, and composite indexes. Status lifecycle: pending → active (on WhatsApp activation). **Architecture hierarchy**: Tenant (email) → Book (join code) → Message → Drops (tags) + Attachments.
- **WhatsApp Integration**: Twilio-based messaging integration using WhatsApp Business API. **Join-code-first routing architecture**: Every incoming message is parsed for a join code, which is the unique routing key. Single O(1) registry lookup by join code determines routing (no phone_to_tenant lookups). **Three routing cases**: (1) Join code found with status='pending' → activate book and set phone_number; (2) Join code found with status='active' → forward message to book's Discord thread; (3) No valid join code → route to limbo (t1-b1). **Phone recycling enabled**: Same phone number can activate multiple books across different tenants because routing is join-code-based, not phone-based. Phone number is just metadata stored in registry, not a routing key. **Sybil-proof activation** via unique join codes (format: "BOOKNAME-abc123"). Each book generates a 6-char random code (16.7M combinations) stored in registry with status='pending'. **Case-preserved parsing** ensures exact matching.
- **Webhook Integration**: Messages are permanently saved to an internal Ledger. Optional personal webhooks (Discord, Slack, etc.) can mirror messages in parallel, with improved UX making webhooks optional for initial book creation. Multi-webhook support allows forwarding to unlimited Discord webhooks simultaneously.
- **Media Handling**: **Discord-native storage** for complete Twilio independence. WhatsApp media is downloaded from Twilio's MediaUrl and uploaded to Discord as native attachments (not embed URLs). Read path uses Toth to fetch Discord attachments, making the system Twilio-independent for reads. Supports all media types (images, videos, audio, documents, Excel files, PDFs, etc.) with automatic file extension detection. Frontend downloads from Discord CDN, ensuring permanent availability. Fallback to embed URLs only if Twilio download fails. **Backward compatibility**: Older messages with embed-only media are parsed via `📎 Media` field (format: `[content-type](url)`).
- **Search & Metadata**: Enhanced search across messages and metadata, supporting multilingual text and full-text search indexing.
- **Real-time Updates**: Smart polling with `?after={messageId}` fetches new messages, pausing when the tab is hidden. Auto-scrolls only if the user is at the bottom, otherwise shows a "New messages" banner. Jump-to-message functionality with `#msg-{messageId}` allows direct navigation to specific messages.
- **Terminology Refactor**: All internal and external "bridge" terminology has been replaced with "book" for consistent branding and user understanding.

**System Design Choices:**
- **Multi-Tenant Isolation**: Complete data separation between users via database schemas.
- **Zero-Friction Onboarding**: WhatsApp deep link activation (no QR scanning, no manual webhook setup required).
- **Scalability & Recovery**: Designed for Replit Autoscale, with PostgreSQL storing all book state for automatic recovery.
- **Security**: Strict webhook validation, production hardening with JWT security, staggered resource initialization, security headers, and robust audit logging. **Sybil attack prevention** via unique one-time join codes (users cannot claim others' phone numbers).
- **CSP Compliance**: Production-ready Content Security Policy with event delegation and self-hosted libraries.

## Recent Changes (November 2025)
- **Database Migration to Supabase**: Migrated production database from Replit's built-in Neon database to Supabase for better control and scalability. Removed all Neon-specific references from codebase. Updated SSL configuration to be database-agnostic (checks for localhost vs production). Production DATABASE_URL now points to Supabase Transaction Mode pooler for 10,000+ concurrent connections.
- **Jump-to-Message UX Fix**: Fixed jump-to-message functionality to clear search state and reload all messages (not just search results) when jumping to a message from search. Message now centers on screen without highlight animation, providing full context (messages above/below) for better UX. Users can access tags and surrounding messages immediately after jumping.
- **Transaction Mode Migration**: Migrated database connection from Supabase **Session Mode** (3-10 connection limit) to **Transaction Mode** (10,000+ concurrent connections) for production scalability. Changes: (1) Pool config increased to max=20, min=2 connections; (2) All tenant-scoped queries refactored to use explicit `${tenantSchema}.table_name` prefixes instead of `SET search_path`; (3) Tenant middleware releases clients immediately after fetching context (no holding for request lifecycle) to prevent pool deadlock; (4) Database URL appends `?pool_mode=transaction` query parameter. Eliminates all `MaxClientsInSessionMode` errors permanently. Architect-approved production-ready implementation.
- **Phone Routing Recency Fix**: Fixed WhatsApp message routing to always use the **most recently activated book** when same phone number is used across multiple books/tenants. Added `updated_at` column to `core.book_registry` (backfilled with activation timestamps). Phone lookup now uses `ORDER BY updated_at DESC LIMIT 1` to select newest activation. Activation updates both `status='active'` and `updated_at=NOW()`. Prevents messages from routing to old, inactive books when user activates a new book with same phone.
- **Media Pipeline Fix**: Fixed WhatsApp media handling for production use. (1) Removed HTTP Basic Authentication from Twilio media downloads (user disabled auth on Twilio account); (2) Added MIME type → file extension mapping for 15 common types (xlsx, docx, pptx, pdf, jpg, png, gif, webp, mp4, mov, mp3, ogg, opus, zip, txt) to fix broken filenames; (3) Verified end-to-end flow: WhatsApp → Twilio download (no auth) → Discord native attachment upload → Frontend serves from Discord CDN. Tested with Excel files and photos. Architect-approved production-ready implementation.
